如果你想做量化开发，最容易走偏的方式有两种：要么把时间都花在金融名词上，要么把它当成普通后端岗来准备。这个岗位真正看重的是三件事：系统性能、工程可靠性、交易场景理解。准备顺序对了，进步会很快；顺序错了，投入再多也会显得虚。

<!-- more -->

## 先分清：你要准备的是哪一类量化开发

虽然 JD 都写着 Quant Developer，但实际差异很大。

| 方向 | 日常工作 | 常见技术栈 | 准备重点 |
| --- | --- | --- | --- |
| 低延迟交易开发 | 行情接入、订单链路、撮合、风控 | C++ / Linux / 网络 / 并发 | C++ 底层、性能优化、系统设计 |
| 研究平台开发 | 回测、因子计算、数据管道、策略框架 | Python / C++ / SQL | Python 数据工具链、工程化、统计基础 |
| 基础设施开发 | 数据采集、存储、调度、监控、运维 | Python / C++ / Kafka / ClickHouse | 稳定性、分布式系统、数据工程 |

大多数校招和实习岗，本质上会落到前两类。先看清楚目标岗位，再决定主线，否则容易学得很散。

## 准备顺序

我的建议是按这个优先级推进：

1. C++ 和系统基础
2. 算法与数据结构
3. Python 与数据处理
4. 概率统计与交易基础
5. 一个能说透的项目

很多人把第 5 条放在最后，结果临近面试才随便拼一个 demo。量化开发反而应该尽早做项目，因为面试官最容易通过项目判断你有没有真实工程感。

## 一、C++ 和系统基础是主轴

如果目标是私募、券商自营、低延迟方向，C++ 决定上限。不是只会语法，而是要能解释性能和行为。

至少要把下面这些内容讲清楚：

- RAII、拷贝控制、move semantics、左值右值、完美转发
- STL 容器的底层特性，尤其是 `vector`、`unordered_map`、`deque`、`priority_queue`
- 智能指针、对象生命周期、内存对齐、缓存友好布局
- 线程、锁、条件变量、原子变量、memory order
- socket、epoll、mmap、零拷贝、cache line、false sharing
- Linux 常用工具：`gdb`、`perf`、`strace`、`valgrind`、`asan/tsan`

量化开发对 C++ 的要求，通常不是“会不会写”，而是“你知不知道这段代码为什么会慢，为什么会出错，为什么线上会抖动”。

## 二、算法不能只停在刷题

量化开发面试里的算法题，往往比普通后端更强调实现质量和复杂度控制。除了常规题型，还容易出现并发设计、随机采样、系统建模。

高频方向：

- 堆、哈希、二叉树、图、滑动窗口、二分、区间问题
- 生产者消费者、读写锁、线程安全队列
- 水塘抽样、加权随机、概率期望
- LRU、延时队列、订单簿、限流器这类工程实现题

刷题目标不要设成“刷了多少道”，而要设成两件事：

- 看到题能快速判断复杂度和数据结构
- 白板或现场编码时，代码能一次写对七八成

## 三、Python 要够实用

哪怕主岗是 C++，Python 也几乎绕不过去。你不一定需要把 Python 玩到很花，但至少要能拿它快速做数据验证、策略实验和脚本工具。

建议掌握这些内容：

- `numpy`、`pandas` 或 `polars`
- 文件读写、日志、命令行脚本、虚拟环境管理
- SQL 基础查询与简单数据清洗
- 回测指标计算：收益率、Sharpe、最大回撤、换手率

如果你只会写 LeetCode 风格的 Python，而不会处理真实数据，面试时会很快露馅。

## 四、数学和交易知识够用就行，但不能完全没有

量化开发不是量化研究，数学要求通常没有研究岗那么深，但以下内容最好熟：

- 条件概率、期望、方差、协方差
- 线性回归、假设检验、过拟合的基本概念
- 收益率、波动率、Sharpe、最大回撤、滑点、手续费
- 限价单、市价单、订单簿、价格优先和时间优先

这里的重点不是背定义，而是能把这些概念和系统行为连起来。比如：

- 为什么回测必须考虑手续费和滑点
- 为什么订单簿撮合要保证价格优先、时间优先
- 为什么风控要在下单链路上尽量靠前

## 五、项目比“我学过很多”更有说服力

如果只能做一个项目，我建议做一个简化版的量化交易系统，而不是单独做一堆零散组件。

可以从下面这个最小闭环开始：

```text
历史行情回放 -> 策略信号 -> 风控检查 -> 下单模拟 -> 撮合/成交 -> 绩效统计
```

这个项目至少要包含：

- 行情数据读取和事件驱动处理
- 简单策略接口，比如均线交叉或盘口信号
- 下单、撤单、成交回报的状态流转
- 回测结果统计和日志输出
- 单元测试或 benchmark

如果你时间更多，再往上加：

- C++ 实现订单簿
- Python 做分析和可视化
- 异步日志库
- 无锁 ring buffer
- `perf` 或 benchmark 的性能报告

面试官真正喜欢听到的是这种描述：

- 我做了什么
- 这个模块为什么这样设计
- 瓶颈在哪
- 我怎么测过
- 我把什么地方优化了

而不是一句“我写过一个交易系统”。

## 一个务实的 8 周准备方案

| 时间 | 重点 | 产出 |
| --- | --- | --- |
| 第 1-2 周 | C++ 基础复盘，STL，内存模型，Linux 工具 | C++ 笔记 + 30 道中等题 |
| 第 3-4 周 | 并发、网络、性能分析 | 线程安全队列 / 异步日志小模块 |
| 第 5-6 周 | 做量化项目主干 | 回测框架或简化订单簿 |
| 第 7 周 | Python 数据分析、统计指标、交易基础 | 策略分析脚本 + 指标输出 |
| 第 8 周 | 模拟面试、补短板、整理简历 | 项目介绍稿 + 高频题清单 |

每天不用拉满十几个小时，但最好固定三段：

- 1 小时 C++ / 系统
- 1 小时算法
- 1 到 2 小时项目

只刷题不做项目，最后会显得像竞赛选手。只做项目不补基础，追问两轮就会暴露底层不扎实。两条线要同时推进。

## 面试官最想看到什么

量化开发面试里，真正拉开差距的通常不是金融知识，而是下面几件事：

- 你有没有性能意识
- 你能不能解释清楚 trade-off
- 你写过的项目有没有真实数据和指标
- 你是否理解稳定性、异常处理和风控的重要性

比如同样是做一个订单簿，实现出来只是及格；能继续讲清楚数据结构选择、复杂度、内存布局、测试方法和极端场景处理，才会更像一个真正能上手的人。

## 几个常见误区

- 误区 1：先学很多金融知识，再开始写代码
- 误区 2：把量化开发当成普通 CRUD 岗准备
- 误区 3：只刷 LeetCode，不做任何可展示项目
- 误区 4：只会 Python，就直接投低延迟 C++ 岗
- 误区 5：简历只写“参与开发”，没有任何量化结果

量化开发是很典型的工程岗位。你要证明的是“我能把一个交易相关系统做出来，而且知道怎么让它更快、更稳、更可控”。

## 推荐资料

书和资料不需要很多，够用就行：

- 《Effective Modern C++》
- 《C++ Concurrency in Action》
- 《Computer Systems: A Programmer's Perspective》
- 《Python for Data Analysis》
- 《An Introduction to Statistical Learning》

题目平台可以选：

- LeetCode
- Codeforces
- 牛客或公开面经里的 C++ / 并发题

## 最后一句

如果你现在还没开始，最稳的路线不是“先把所有知识点学完”，而是马上做一条主线：

先把 C++ 和系统基础补齐，同时做一个简化版量化项目，再用算法和 Python 把它撑起来。做到这一步，你的准备就已经比大多数“只会背八股”和“只会刷题”的候选人更像真正的量化开发。
